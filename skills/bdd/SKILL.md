# BDD（振る舞い駆動開発）

**鉄則: シナリオの承認を得るまで実装ファイルに触れない。**

## 1. シナリオ定義

正常系・異常系・エッジケース（権限が絡む場合はロール別）を洗い出し、Gherkin 形式でチャットに提示して承認を得る。

```gherkin
Feature: [機能名]

  Scenario: [振る舞いの説明]
    Given [初期状態]
    When  [操作]
    Then  [期待される結果]
```

- 1 シナリオ = 1 振る舞い
- 実装詳細（クラス名・メソッド名）を書かず、ユーザー視点の言葉で書く
- 仕様の曖昧さは推奨案を添えて、この時点でまとめて解消する

## 2. 自動化と実装（Red → Green → Refactor）

承認後は自律的に進める：

1. プロジェクトのテストフレームワークと配置規約を検出し、シナリオを振る舞いに最も近いレイヤのテスト（E2E・受け入れ・結合）として自動化する
2. Red（失敗）を確認 → Green になる最小の実装 → Refactor
