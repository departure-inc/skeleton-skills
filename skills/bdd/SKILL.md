# BDD（振る舞い駆動開発）

## 鉄則

**振る舞いシナリオを定義する前に実装を始めてはならない。**

---

## フェーズ 1: Discovery（Three Amigos）

- 実装する機能の「誰が・何をしたとき・どうなるか」を整理する
- 以下の観点でシナリオを洗い出す：
  - 正常系（ハッピーパス）
  - 異常系・エッジケース
  - 権限・認証が絡む場合はロール別の挙動
- GitHub Issue の Acceptance Criteria があれば突き合わせる

---

## フェーズ 2: Formulation（シナリオ定義）

以下の形式でシナリオを記述し、チャットに出力してユーザーに確認する：

```gherkin
Feature: [機能名]

  Scenario: [正常系の説明]
    Given [初期状態]
    When  [ユーザーの操作]
    Then  [期待される結果]

  Scenario: [異常系の説明]
    Given ...
    When  ...
    Then  ...
```

**シナリオ記述ルール:**
- 1 シナリオ = 1 振る舞い（複数の振る舞いを混在させない）
- 実装の詳細（メソッド名・クラス名）を書かない
- ユーザー視点の言葉を使う

---

## フェーズ 3: Automation（TDD で実装）

ユーザーの承認を得てから：

1. シナリオを RSpec feature spec / system spec として自動化する（保存先: `spec/features/` または `spec/system/`）
2. スペックが Red（失敗）であることを確認する
3. Green（通過）になるよう実装する
4. Refactor して整理する

---

## 注意事項

- シナリオの承認を得るまで実装ファイルには触れない
- 曖昧な仕様は Discovery フェーズでユーザーに一問ずつ確認する
