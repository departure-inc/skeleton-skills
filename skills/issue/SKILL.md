ユーザーと壁打ちして機能設計を詰め、合意した内容を GitHub ISSUE として作成する。

一問ずつユーザーとやり取りする対話なのでサブエージェントには委譲しない。kabe から続けて呼ばれた場合はセッションのモデルをそのまま引き継ぐ。

## 1. 壁打ち

「どんな機能を作りたいですか？」から始め、**一問ずつ・必ず推奨回答付き**で設計を詰める。

- 先にプロジェクトの技術スタックと既存コードを調べ、質問と用語をそのスタックに合わせる
- コードベースを調べれば分かることは質問しない

確認する観点（機能に応じて取捨選択）：
WHY（解決する課題）／データ設計（構造・型・制約）／ドメインロジック／API・ルーティング／認証・認可／UI／非同期処理・外部連携／削除時の連鎖挙動／完了条件

## 2. ISSUE 本文生成

合意が取れたら GFM で本文を生成する。不要なセクションはスキップする。

```
# WHY
[課題と価値]

# WHAT
* [ ] [実装タスクのチェックリスト]

# 設計
[概要／データ設計／API／UI など、壁打ちで合意した設計判断。
コードは方針が伝わる最小限にとどめる]

# 完了条件
[何ができれば実装完了か]
```

## 3. ISSUE 作成

本文を出力後「このまま ISSUE を作成しますか？」と確認し、許可を得てから実行する：

```bash
gh issue create --title "[機能名]" --body "[本文]" --label "feature"
```
